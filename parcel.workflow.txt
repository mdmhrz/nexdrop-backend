# Parcel Module - Brief Summary

The parcel module enables customers to create delivery requests, riders to accept and deliver parcels, and admins to monitor and manage the system.

**Current State**: All core parcel workflow APIs are implemented without payment integration.
- Customers can create parcels
- Riders can accept/deliver parcels
- Admins can assign riders and override statuses
- Parcel status tracking with logs is fully functional
- Rider earnings are generated but not yet paid

**Known Limitation**: Payment processing is not implemented. The `isPaid` field remains `false` and rider earnings stay in `PENDING` status indefinitely. Payment integration will be added tomorrow.

---

Complete Parcel Workflow
Phase 1: Customer Creates Parcel
API: POST /parcels (CUSTOMER only)

Customer submits parcel with:
Pickup address
Delivery address
District from/to
Price
Optional note
System:
Generates unique tracking ID
Creates parcel with status = REQUESTED
Creates status log
No rider assigned (riderId = null)
Phase 2: Rider Assignment (Two Ways)
Option A: Admin Manual Assignment API: PATCH /parcels/:id/assign-rider (ADMIN, SUPER_ADMIN only)

Admin selects parcel and assigns a rider
System validates:
Parcel status is REQUESTED
Rider exists with ACTIVE account status
Rider is AVAILABLE (not BUSY or OFFLINE)
System updates:
Parcel status → ASSIGNED
Rider assigned to parcel
Status log created
Option B: Rider Self-Assignment

Step 1: Browse Available Parcels
API: GET /parcels/available (RIDER, ADMIN, SUPER_ADMIN)
- Riders must have ACTIVE account status
- Admins and Super Admins can access without rider account status check
- Shows parcels with status = REQUESTED, riderId = null

Step 2: Accept/Claim Parcel
API: PATCH /parcels/:id/accept (RIDER only - ACTIVE riders)

Rider accepts/claims a parcel
System validates:
Parcel status is REQUESTED
Rider is ACTIVE and AVAILABLE
Parcel not already assigned
System updates:
Parcel status → ASSIGNED
Rider assigned to parcel
Status log created
Phase 3: Rider Views Assigned Parcels
API: GET /parcels/assigned (RIDER only - ACTIVE riders)

Rider views all parcels assigned to them
Shows parcels with status: ASSIGNED, IN_TRANSIT
Phase 4: Rider Picks Up Parcel
API: PATCH /parcels/:id/pick (RIDER only - ACTIVE riders)

Rider goes to pickup location
Marks parcel as picked (auto-transits to IN_TRANSIT)
System validates:
Parcel is assigned to this rider
Parcel status is ASSIGNED
Rider is ACTIVE
System updates:
Parcel status → IN_TRANSIT
Rider status → BUSY
Status log created
Phase 5: Rider Delivers Parcel
API: PATCH /parcels/:id/deliver (RIDER only - ACTIVE riders)

Rider completes delivery
Marks parcel as delivered
System validates:
Parcel is assigned to this rider
Parcel status is IN_TRANSIT
Rider is ACTIVE
System updates:
Parcel status → DELIVERED
Rider status → AVAILABLE
Status log created
Earning generated (70% of parcel price, status = PENDING)
Phase 6: Customer Tracks Parcel
API: GET /parcels/:id (CUSTOMER, RIDER only)

Customer views parcel details
Shows current status, rider info, status logs with timestamps
Phase 7: Customer Views Their Parcels
API: GET /parcels/my (CUSTOMER only)

Customer views all their parcels (any status)
Paginated list with rider info
Phase 8: Customer Cancels Parcel
API: PATCH /parcels/:id/cancel (CUSTOMER only)

Customer can cancel only if status = REQUESTED
System validates:
Parcel belongs to customer
Status is REQUESTED
System updates:
Parcel status → CANCELLED
Status log created
Admin Monitoring & Management
View All Parcels API: GET /parcels (ADMIN, SUPER_ADMIN only)

Filter by status, district, date
Monitor all parcels across the system
View Available Parcels API: GET /parcels/available (RIDER, ADMIN, SUPER_ADMIN)

Riders: Only if ACTIVE account status
Admins: No account status check
Shows parcels with status = REQUESTED, riderId = null
Override Status API: PATCH /parcels/:id/status (ADMIN, SUPER_ADMIN only)

Admin can manually change any parcel status
Creates status log
Rider Discovery
Browse Available Jobs API: GET /parcels/available (RIDER, ADMIN, SUPER_ADMIN)

Riders see parcels they can accept
Admins see all available parcels for assignment
Parcel Status Flow
REQUESTED → ASSIGNED → IN_TRANSIT → DELIVERED
                    ↓
                 CANCELLED
Status Meanings:

REQUESTED: Customer created parcel, waiting for rider assignment
ASSIGNED: Rider assigned, waiting for pickup
IN_TRANSIT: Rider picked up and is en route to delivery
DELIVERED: Successfully delivered, earning generated
CANCELLED: Cancelled by customer (only in REQUESTED status)
Rider Status Flow
AVAILABLE → BUSY → AVAILABLE
Status Meanings:

AVAILABLE: Rider can accept new parcels
BUSY: Rider is on a delivery (after picking up parcel)
OFFLINE: Rider not accepting deliveries
Complete Example Flow
Customer:

POST /parcels → Creates parcel (status: REQUESTED)
Admin (or Rider): 2. PATCH /parcels/:id/assign-rider → Assigns rider (status: ASSIGNED) OR PATCH /parcels/:id/accept → Rider accepts (status: ASSIGNED)

Rider: 3. GET /parcels/assigned → Views assigned job 4. PATCH /parcels/:id/pick → Picks up parcel (status: PICKED, rider: BUSY) 5. PATCH /parcels/:id/deliver → Delivers parcel (status: DELIVERED, rider: AVAILABLE, earning generated)

Customer: 6. GET /parcels/:id → Tracks parcel progress 7. GET /parcels/my → Views all parcels
