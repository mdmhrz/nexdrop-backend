# Parcel Module - Brief Summary

The parcel module enables customers to create delivery requests, riders to accept and deliver parcels, and admins to monitor and manage the system.

**Current State**: All core parcel workflow APIs are implemented WITH payment integration.
- Customers can create parcels
- Riders can accept/deliver parcels
- Admins can assign riders and override statuses
- Parcel status tracking with logs is fully functional
- Payment processing implemented (Stripe & SSL Commerz)
- Rider earnings generated on successful payment

---

Complete Parcel Workflow
Phase 1: Customer Creates Parcel
API: POST /parcels (CUSTOMER only)

Customer submits parcel with:
Pickup address
Delivery address
District from/to
Price
Optional note
System:
Generates unique tracking ID
Creates parcel with status = REQUESTED
Creates status log
No rider assigned (riderId = null)
Phase 2: Rider Assignment (Two Ways)
Option A: Admin Manual Assignment API: PATCH /parcels/:id/assign-rider (ADMIN, SUPER_ADMIN only)

Admin selects parcel and assigns a rider
System validates:
Parcel status is REQUESTED
Rider exists with ACTIVE account status
Rider is AVAILABLE (not BUSY or OFFLINE)
System updates:
Parcel status → ASSIGNED
Rider assigned to parcel
Status log created
Option B: Rider Self-Assignment

Step 1: Browse Available Parcels
API: GET /parcels/available (RIDER, ADMIN, SUPER_ADMIN)
- Riders must have ACTIVE account status
- Admins and Super Admins can access without rider account status check
- Shows parcels with status = REQUESTED, riderId = null

Step 2: Accept/Claim Parcel
API: PATCH /parcels/:id/accept (RIDER only - ACTIVE riders)

Rider accepts/claims a parcel
System validates:
Parcel status is REQUESTED
Rider is ACTIVE and AVAILABLE
Parcel not already assigned
System updates:
Parcel status → ASSIGNED
Rider assigned to parcel
Status log created
Phase 3: Rider Views Assigned Parcels
API: GET /parcels/assigned (RIDER only - ACTIVE riders)

Rider views all parcels assigned to them
Shows parcels with status: ASSIGNED, IN_TRANSIT
Phase 4: Rider Picks Up Parcel
API: PATCH /parcels/:id/pick (RIDER only - ACTIVE riders)

Rider goes to pickup location
Marks parcel as picked (auto-transits to IN_TRANSIT)
System validates:
Parcel is assigned to this rider
Parcel status is ASSIGNED
Rider is ACTIVE
System updates:
Parcel status → IN_TRANSIT
Rider status → BUSY
Status log created
Phase 5: Rider Delivers Parcel
API: PATCH /parcels/:id/deliver (RIDER only - ACTIVE riders)

Rider completes delivery
Marks parcel as delivered
System validates:
Parcel is assigned to this rider
Parcel status is IN_TRANSIT
Rider is ACTIVE
System updates:
Parcel status → DELIVERED
Rider status → AVAILABLE
Status log created
Earning generated (70% of parcel price, status = PENDING)
Phase 6: Customer Tracks Parcel
API: GET /parcels/:id (CUSTOMER, RIDER only)

Customer views parcel details
Shows current status, rider info, status logs with timestamps
Phase 7: Customer Views Their Parcels
API: GET /parcels/my (CUSTOMER only)

Customer views all their parcels (any status)
Paginated list with rider info
Phase 8: Customer Cancels Parcel
API: PATCH /parcels/:id/cancel (CUSTOMER only)

Customer can cancel only if status = REQUESTED
System validates:
Parcel belongs to customer
Status is REQUESTED
System updates:
Parcel status → CANCELLED
Status log created
Admin Monitoring & Management
View All Parcels API: GET /parcels (ADMIN, SUPER_ADMIN only)

Filter by status, district, date
Monitor all parcels across the system
View Available Parcels API: GET /parcels/available (RIDER, ADMIN, SUPER_ADMIN)

Riders: Only if ACTIVE account status
Admins: No account status check
Shows parcels with status = REQUESTED, riderId = null
Override Status API: PATCH /parcels/:id/status (ADMIN, SUPER_ADMIN only)

Admin can manually change any parcel status
Creates status log
Rider Discovery
Browse Available Jobs API: GET /parcels/available (RIDER, ADMIN, SUPER_ADMIN)

Riders see parcels they can accept
Admins see all available parcels for assignment
Parcel Status Flow
REQUESTED → ASSIGNED → IN_TRANSIT → DELIVERED
                    ↓
                 CANCELLED
Status Meanings:

REQUESTED: Customer created parcel, waiting for rider assignment
ASSIGNED: Rider assigned, waiting for pickup
IN_TRANSIT: Rider picked up and is en route to delivery
DELIVERED: Successfully delivered, earning generated
CANCELLED: Cancelled by customer (only in REQUESTED status)
Rider Status Flow
AVAILABLE → BUSY → AVAILABLE
Status Meanings:

AVAILABLE: Rider can accept new parcels
BUSY: Rider is on a delivery (after picking up parcel)
OFFLINE: Rider not accepting deliveries
Complete Example Flow
Customer:

POST /parcels → Creates parcel (status: REQUESTED)
Admin (or Rider): 2. PATCH /parcels/:id/assign-rider → Assigns rider (status: ASSIGNED) OR PATCH /parcels/:id/accept → Rider accepts (status: ASSIGNED)

Rider: 3. GET /parcels/assigned → Views assigned job 4. PATCH /parcels/:id/pick → Picks up parcel (status: PICKED, rider: BUSY) 5. PATCH /parcels/:id/deliver → Delivers parcel (status: DELIVERED, rider: AVAILABLE, earning generated)

Customer: 6. GET /parcels/:id → Tracks parcel progress 7. GET /parcels/my → Views all parcels

## Code Quality & Refactoring TODO

### APIs That Need Better Implementation

#### 1. Parcel Status Update Services (HIGH PRIORITY)
**Files**: `acceptParcel.service.ts`, `pickParcel.service.ts`, `deliverParcel.service.ts`, `assignRider.service.ts`, `cancelParcel.service.ts`

**Issues**:
- Repetitive rider validation logic (find rider, check accountStatus, check currentStatus)
- Repetitive parcel validation logic (find parcel, check status, check ownership)
- Repetitive transaction pattern (update parcel + create status log)
- Status log creation is duplicated in every service

**Solution**: Create a shared `ParcelStatusService` utility with methods like:
- `validateRiderAccess(riderId, requiredStatus)`
- `validateParcelAccess(parcelId, userId, requiredStatus)`
- `updateParcelStatus(parcelId, newStatus, changedBy, note)`

#### 2. Rider Earnings Service (MEDIUM PRIORITY)
**File**: `riderEarnings.service.ts`

**Issues**:
- Uses `throw new Error()` instead of `AppError` (inconsistent error handling)
- Uses `any` type for query parameters (not type-safe)

**Solution**:
- Replace `throw new Error()` with `AppError(status.NOT_FOUND, "message")`
- Define proper TypeScript interface for query parameters

#### 3. Dashboard Services (MEDIUM PRIORITY)
**Files**: `riderDashboard.service.ts`, `userDashboard.service.ts`

**Issues**:
- Duplicate average delivery time calculation logic
- Duplicate last 7 days data generation logic
- Both have nearly identical structure

**Solution**: Create shared utility functions:
- `calculateAverageDeliveryTime(parcels)`
- `generateTimeSeriesData(entityId, days, aggregationFn)`

#### 4. Stats Cache Service (LOW PRIORITY)
**File**: `statsCache.service.ts`

**Issues**:
- Initialization code duplicated in `updateStatsCache` and `getStatsCache`
- No race condition protection for concurrent updates
- Magic numbers scattered throughout

**Solution**:
- Extract initialization to a single `initializeCache()` function
- Add optimistic locking or use database transactions
- Move constants to config file

### Code Repetition - Areas to Reduce

#### 1. Rider Validation Pattern (Repeated 4+ times)
```typescript
// Repeated in: acceptParcel, pickParcel, deliverParcel
const rider = await prisma.rider.findUnique({ where: { userId: riderId } });
if (!rider) throw AppError(...);
if (rider.accountStatus !== RiderAccountStatus.ACTIVE) throw AppError(...);
```

**Solution**: Create `src/app/shared/services/riderValidation.service.ts`

#### 2. Parcel Status Update Pattern (Repeated 5+ times)
```typescript
// Repeated in: acceptParcel, pickParcel, deliverParcel, assignRider, cancelParcel
await prisma.$transaction(async (tx) => {
  await tx.parcel.update(...);
  await tx.parcelStatusLog.create(...);
});
```

**Solution**: Create `src/app/shared/services/parcelStatus.service.ts`

#### 3. Stats Cache Initialization (Duplicated)
**Solution**: Extract to single function with default values object

### Non-Industry Standard Practices

#### 1. Inconsistent Error Handling
- Some services use `throw new Error()` (riderEarnings.service.ts)
- Others use `AppError` with proper status codes
- **Fix**: Standardize all services to use `AppError`

#### 2. Type Safety Issues
- `query: any` in earnings service
- No proper interfaces for query parameters
- **Fix**: Define Zod schemas or TypeScript interfaces for all query params

#### 3. Magic Numbers
- `0.7` for rider percentage (should be in config)
- `take: 5`, `take: 100` limits (should be constants)
- **Fix**: Move to `src/app/config/constants.ts`

#### 4. No Constants File
- Hardcoded values scattered throughout
- **Fix**: Create constants file for:
  - Rider commission percentage
  - Pagination limits
  - Time limits (24h for rating edits)
  - Cache TTL values

#### 5. Potential Race Conditions
- Stats cache updates without locking
- Multiple concurrent updates could overwrite each other
- **Fix**: Use Prisma transactions or optimistic locking

#### 6. N+1 Query Risk
- Dashboard services fetch parcels then fetch status logs separately
- **Fix**: Use Prisma `include` to fetch related data in single query

### Recommended Refactoring Priority

1. **HIGH**: Extract rider validation to shared service
2. **HIGH**: Extract parcel status update pattern to shared service
3. **MEDIUM**: Fix error handling inconsistency (replace `Error` with `AppError`)
4. **MEDIUM**: Add type safety to query parameters
5. **LOW**: Create constants file
6. **LOW**: Add race condition protection to stats cache
